# プロジェクト変更・修正ポリシー

研究用シミュレーション結果の再利用性を守りつつ拡張性を維持するためのルールである。自律的なリファクタや「きれいにする」変更にも適用する。**リポジトリオーナーが明示的に指示したタスク**は、このポリシーの制限を緩和してよい（その場合でも成果物の版分けは推奨する）。

---

## 1. 目的

- これまで積み上げた**シミュレーション結果の再利用可能性**（同じ CSV・図・論文用数値が再現・比較できること）を最優先する。
- **拡張**（新 policy・新 study・新 CLI オプション）は歓迎するが、**既存成果物を無効化する変更**は原則避ける。

---

## 2. 変更のレイヤー分け

| レイヤー | 例 | 原則 |
|----------|-----|------|
| **A. 研究アーティファクト** | `results/studies/*/summary/*.csv`、確定済みの図・表 | 上書きで意味が変わる変更はしない。再実験は **新ディレクトリ / 新 study_id** とする。 |
| **B. コア契約** | sweep 出力列名・型、aggregate のキー、artifact JSON スキーマ、メトリクス定義 | 後方互換のため **追加のみ** または **バージョン付き**（`schema_version` 等）。**列の削除・意味変更・単位変更は原則禁止**。 |
| **C. スケジューラ実装** | `scheduler.py`、issue / ff / meas のサイクル進行ロジック | **挙動変更は原則禁止**。バグ修正のみ可とする場合は、既存 canonical データとの差分を記録する。 |
| **D. 拡張ポイント** | 新 `Policy`、新 CLI サブコマンド、新 `results/studies/<new_id>/` | **自由**。既存 policy 名・既存デフォルトの挙動を変えない限りよい。 |

---

## 3. 事前合意なく進めてよい想定の変更

- **新しい policy** の追加（既存 enum の値追加、新しいスケジューラ実装の追加）。
- **新しい study** 用の出力先ディレクトリ・manifest・sweep パラメータ。
- **ドキュメント**（`docs/worklog`、`docs/docs_old`、レポート）の追記・整理。
- **分析・プロット用スクリプト**で、入力 CSV の列を**読むだけ**の追加可視化。

---

## 4. 慎重に扱う変更（原則「やらない」か、やるなら手順付き）

- 既存 **policy のスケジューリング順序・閾値・トリガー条件**の変更。
- **デフォルト引数**（release_mode、幅、latency）の変更。
- **列名・集約キー・比較ロジック**（paired comparison の定義）の変更。
- **同じ study パス**への再 sweep で、アルゴリズム・シード以外が変わるようなコード変更。

---

## 5. やむを得ずコア（B / C）を変える場合の最低条件

1. **新しい study_id** または `rerun_YYYYMMDD` 相当で結果を分離する。
2. **CHANGELOG または worklog** に「何が変わり、どのデータが旧版と非比較か」を一文以上書く。
3. 可能なら **旧コードをタグまたはブランチ**で残す、または **manifest に simulator の git SHA** を載せる。

---

## 6. 運用上のまとめ

- **Policy の追加は OK**。**それ以外のコア（スキーマ・スケジューラ・既存 study の数値）**は極力変更しない。
- エージェントや共同作業者向けの目安: 触ってよいのは主に **D（拡張ポイント）** と **新規 `results/studies/<...>`**。

# LatencyPipeline の改善方針メモ

## 位置づけ

`LatencyPipeline` は、現状では「固定 latency・固定 width を持つ簡易パイプラインモデル」であり、シミュレーションを始めるための最小実装として有用である。  
一方で、現実の MBQC classical control を考えるにはかなり単純化されているため、今後はより実機制約を反映したモデルへ拡張していく余地が大きい。

---

## 現状モデルの特徴

- 各 item は `node_id` のみで表現される
- latency は全 item 共通の固定値
- width は 1 サイクルあたりの受け入れ数上限としてのみ働く
- fully pipelined を仮定しており、毎サイクル投入可能
- ステージ内部での競合や動的遅延は考慮しない

---

## 改善の大きな方向性

### 1. latency を固定値から拡張する

現状では全 node が同じ latency を持つが、実際には処理内容によって遅延は異なりうる。  
そのため、以下のような拡張が考えられる。

- node type ごとに latency を変える
- dependency の種類ごとに latency を変える
- decoder 呼び出しの有無で latency を変える
- 混雑状況や資源競合によって latency が増加するようにする

#### ねらい

より現実的な classical control の時間モデルを表現すること。

---

### 2. width を単なる本数制限ではなく資源制約として扱う

現状の `width` は「1 サイクルに何個入るか」のみを表している。  
しかし実際には、制約は単純な本数だけではなく、共有資源や競合として現れる可能性が高い。

例えば以下を導入できる。

- measurement lane 数の制約
- decoder の共有利用による競合
- feed-forward 処理器の同時実行数制限
- basis switching に伴う追加コスト
- 入出力バッファ容量の制約

#### ねらい

単純なスループット制限ではなく、資源競合に起因するボトルネックを表現すること。

---

### 3. item を `node_id` だけでなく属性付きにする

現状では pipeline に流れるのは単なる `node_id` であり、処理に必要な追加情報を持たない。  
今後は以下のような情報を持つ token に拡張することが考えられる。

- node type
- dependency type
- 優先度
- 必要資源種別
- 発行時刻
- ready になった理由や依存解消状況

#### ねらい

node ごとの差異を pipeline モデルに持ち込みやすくすること。

---

### 4. MBQC 固有の制約を導入する

一般的な pipeline モデルにとどまらず、MBQC 特有の制約を反映させることで研究的価値が高まる。

候補としては以下がある。

- 測定結果到着タイミングのばらつき
- feed-forward の依存解消待ち
- signal shifting 前後での制御負荷差
- photonic 系における delay line の保持時間制約
- resource state の寿命制約
- fusion 成否に応じた動的再スケジューリング

#### ねらい

単なる一般パイプラインではなく、MBQC classical control 専用の評価モデルに近づけること。

---

### 5. パイプライン内部挙動をより現実的にする

現状では、各サイクルで一律に 1 段ずつ進むだけのきれいな pipeline になっている。  
今後は以下のような挙動も検討できる。

- 一部ステージでストールが起きる
- downstream の詰まりによる backpressure
- 内部バッファが満杯になったときの停止
- item ごとの処理順変更
- flush や bubble の挿入

#### ねらい

理想的な fully pipelined モデルから、よりマイクロアーキテクチャ寄りのモデルに発展させること。

---

## 段階的な改善案

いきなり複雑化しすぎると比較評価が難しくなるため、以下のように段階的に拡張するのがよさそうである。

### Phase 1: baseline の明確化

- 現状の fixed latency / fixed width モデルを baseline として整理する
- 指標として makespan, utilization, stall 率などを定義する

### Phase 2: node-dependent latency

- node type や dependency type に応じて latency を変える
- 固定モデルとの差を比較する

### Phase 3: resource contention

- shared decoder や lane 数制限を導入する
- width モデルとの差を比較する

### Phase 4: MBQC-specific constraints

- signal shifting, feed-forward, delay line 制約などを導入する
- MBQC 固有条件下で支配的なボトルネックを調べる

---

## 評価観点

改善後のモデルは、単に複雑にするだけでなく、結果への影響を評価できる形にしたい。

主な評価観点は以下。

- makespan がどれだけ変わるか
- どの制約が律速になるか
- fixed latency モデルではどこまで十分か
- signal shifting や resource sharing が制御遅延にどう効くか
- hardware size や DAG 構造によって傾向がどう変わるか

---

## 研究テーマとしての見どころ

この部分は単なる実装詳細ではなく、 classical control の評価モデルそのものになりうる。  
特に以下の問いを立てられると研究として強い。

- どの制約を入れると評価結果が本質的に変わるか
- 単純な fixed latency モデルはどの条件で妥当か
- MBQC 固有制約は classical control cost にどの程度影響するか
- signal shifting や architecture choice が制御ボトルネックをどう変えるか

---

## 一言まとめ

`LatencyPipeline` は現状では「最小の時間遅延モデル」だが、  
ここに node 依存 latency・資源競合・MBQC 固有制約を段階的に導入していくことで、  
classical control architecture の評価モデルとして大きく発展させられる。

## 実験目的

MBQC classical control において、

1. shifted DAG 化が制御レイテンシに与える影響
2. measurement/FF 資源制約が makespan に与える影響
3. scheduler の有効性が現れる regime

を明らかにする。

## 実験1: raw vs shifted

- policies: asap, greedy_critical
- issue_widths: 4,8
- l_meas: 1,2
- l_ff: 1,2
- meas_widths: 4,8
- ff_widths: 4,8

- release_mode: next_cycle
- dag_variant: both

評価:

- total_cycles
- stall_rate
- utilization
- ff_chain_depth
- shifted/raw 改善率

## 実験2: bottleneck regime characterization

- policy: asap, shifted_critical, stall_aware_shifted
- regime A: meas bottleneck (meas_width < ff_width)
- regime B: ff bottleneck (ff_width < meas_width)
- regime C: balanced
- issue_width: 8 固定

評価:

- total_cycles
- stall_rate
- ff_waiting_queue_size
- in_flight_meas / in_flight_ff
- regime ごとの律速要因

## 実験3: scheduler comparison

- policies: asap, greedy_critical, shifted_critical, stall_aware_shifted, regime_switch, regime_switch_refined
- 実験2で選んだ代表 regime で比較
- dag_variant: raw, shifted

評価:

- makespan 改善率
- stall_rate 改善率
- regime 別勝者
- shifted DAG との相性

## 仮説

- H1: shifted DAG は average makespan を短縮する
- H2: FF bottleneck では stall-aware 系が ASAP を上回る
- H3: fully provisioned 領域では ASAP と高度 scheduler の差は小さい
- H4: regime-switch 系は複数 regime をまたぐ条件で安定に高性能
